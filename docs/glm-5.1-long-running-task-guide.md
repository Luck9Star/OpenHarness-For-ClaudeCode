# 8 小时自主工作靠什么实现？一文讲清 GLM-5.1 长程任务的实操方法

来源: https://zhipu-ai.feishu.cn/wiki/VlW8w8fuZisEThkaDchcmMaRnNf

---

GLM-5.1 最受关注的亮点之一，是它可以在单次任务中持续、自主地工作长达 8 小时。
但对很多开发者来说，真正关心的并不是"8 小时"这个数字本身，而是更实际的问题：
- 这样的长程任务到底怎么跑起来？
- 开发中又该怎么真正把它用起来？

这也是这篇文章想讨论的重点。
接下来，我们就从这几个问题展开：什么样的任务适合长程执行，8 小时任务通常是怎么推进的，以及在实际落地时需要准备哪些基础配置。

## 一、什么叫"长程任务"？

很多人会把长程任务理解成"模型能聊更久"或者"上下文更长"，但这并不准确。对开发者来说，真正的长程任务通常有几个共同特征：目标复杂、步骤很多、执行过程中会不断遇到新信息，还需要一边做一边验证，再根据结果继续调整。 也就是说，它不是一次性生成，而是一个持续拆解、持续执行、持续修正的过程。

从这个角度看，长程任务的关键在于能不能在很长一段执行链路里，始终围绕同一个目标稳定推进。

## 二、哪些场景适合用长程任务？

并不是所有需求都值得用长程模式来跑。真正适合长程跑的，通常是那些目标明确，但完成路径较长的任务：它们往往需要跨文件、跨模块、跨工具持续推进，中途还要不断读取上下文、执行命令、检查结果、修复问题，最后一步步逼近可交付结果。

换句话说，长程任务更适合那些天然需要"计划—执行—验证—修复—继续推进"闭环的工程型工作，而不是一次性生成就能结束的简单需求。

比较典型的场景包括：

### 接手一个陌生项目，并在理解代码库后继续开发
比如先梳理系统结构、关键模块和数据流，再定位相关文件，最后在已有架构上新增一个功能。这类任务不是简单"写一段新代码"，而是要先理解全局，再逐步落到具体实现上。

### 从一个线上问题或报错出发，持续定位并修复 bug
比如根据报错日志、复现步骤和调用链，逐步查出真正原因，再修改代码、回归验证，必要时继续追踪连带问题。这类任务往往不是一次修改就结束，而是要经历多轮排查与修复。

### 对老项目做大规模重构或技术迁移
比如清理 deprecated API、把旧写法迁到新框架版本、把零散逻辑改成更现代的工程模式，同时保持行为一致并持续跑测试验证。这种任务的核心不是"重写"，而是在不破坏现有系统的前提下，分阶段推进改造。

### 补测试、提覆盖率，并顺手修掉暴露出来的问题
比如先找出未覆盖模块，再生成测试脚手架，补 edge cases，跑测试后继续修复失败用例，直到覆盖率和稳定性都达标。它很适合长程任务，因为"写测试"本身只是开始，真正耗时的是后面的运行、定位和修复。

### 从 issue / PR / 安全告警出发，自动完成一整轮修改并产出 PR
比如把一个 issue 直接交给 agent，让它在仓库里完成实现并开 PR；或者在现有 PR 上继续处理 review feedback；再或者根据安全告警去修补漏洞。这类场景很适合长程执行，因为它天然有明确输入、可跟踪进度和清晰交付物。

它们的共同点不是"任务大"，而是任务本身就需要多轮推进，并且每一轮都有新的上下文和反馈进入系统。在这类任务里，模型的价值不在于"一次写完"，而在于它能在较长时间里持续保持目标一致性，把一系列零散动作串成一个真正可交付的工程闭环。

## 三、8 小时任务一般怎么具体操作？

### 3.1 先把目标写成"可交付结果"，不要只给愿景

长程任务的第一步是先把任务目标写清楚。这里最容易犯的错误，是给模型一个很大的愿景，但没有给出清晰的交付标准。

比如"帮我做一个 Linux 桌面"这种表述，听起来很完整，但对执行系统来说其实过于模糊。更好的写法应该是：最终要交付什么、至少要包含哪些模块、哪些能力必须可运行、什么情况下算完成、哪些属于加分项。任务定义越清楚，后面的长程执行越稳定。

比如，把目标写成：
- 产出一个可运行的 Linux 桌面环境
- 包含窗口管理器、状态栏、基础应用、字体支持、VPN 管理器
- 可以打包并运行
- 有测试或最小验收脚本
- 最终输出源码、运行说明和关键架构说明

这样就比一句"做个桌面系统"强得多。

### 3.2 把任务拆成阶段，而不是一次性做完

第二步，是把任务拆成几个可连续推进、可单独验收的阶段。长程任务最怕的不是任务太大，而是没有阶段性目标。一旦没有里程碑，模型在很长的执行过程中就容易出现策略漂移：明明一直在工作，但产出并不往最终目标上靠。

更有效的做法，是先把任务拆成几段明确的执行路径，并且让每一段都有清晰产物。例如：

![五阶段流程图](whiteboard-2.png)

1. **第一阶段：搭骨架** — 建立项目结构、技术选型、核心框架
2. **第二阶段：跑通最小版本** — 实现最小可运行原型，验证基本功能
3. **第三阶段：逐步补全核心能力** — 按模块逐步添加功能
4. **第四阶段：补测试与修问题** — 完善测试覆盖、修复已知问题
5. **第五阶段：整理和交付** — 文档、收敛、最终验证

这样拆分的好处是，模型每一轮都知道自己当前处于哪个阶段、这一阶段的目标是什么、完成标志是什么。

对于长程任务来说，阶段拆分的意义是让模型始终围绕当前子目标推进，并在阶段之间逐步逼近最终交付结果。

### 3.3 状态不要全放在上下文里，要写到外部

第三步是把状态外置。真正稳定的做法，通常不是把所有信息都塞进一次会话，而是把目标、待办、当前进展、已知问题、验收标准这些内容写到外部文件里，让每一轮都可以重新读取。Claude Code 官方就明确说明，每个会话都从新的 context window 开始，而跨会话延续知识主要靠 CLAUDE.md 这类持久指令和 auto memory；外部 loop 工具 Ralph 的思路也很类似：每次迭代都以新的上下文启动，但状态通过 git、progress.txt、prd.json 这类文件持续保留。

建议至少准备这些文件：

- **goal.md**：任务目标和边界
- **todo.md**：任务拆解和当前状态
- **progress.md**：每轮做了什么、还卡在哪
- **acceptance.md**：验收标准
- **run.sh / test.sh**：一键运行或验证脚本

长程任务真正稳定的记忆，不在模型脑子里，而在 repo 里。

### 3.4 每一轮都必须"做事 + 验证 + 记录"

第四步，是让模型每一轮都必须经历一个最小闭环：先选一个当前子任务，接着执行代码或命令，再验证结果，最后把进展写回状态文件。 这一步很重要，因为长程任务最危险的情况，不是停下来，而是"看起来一直在忙，但没有形成可积累的进展"。

每一轮最好都要求模型回答四个问题：
- 当前子任务是什么
- 这轮准备怎么做
- 怎么判断这轮做成了
- 结果要更新到哪个状态文件里

这会让整个过程明显稳定很多。

### 3.5 最后才是把 loop 跑起来

做完前面的准备，才轮到真正启动 loop。loop 的作用是让模型不会在一轮结束后就停下，而是继续往目标推进。 如果你已经在用 Claude Code，官方本身就提供 /loop 作为 CLI 会话里的重复提示机制；如果你想把这件事做得更自动化，也可以考虑 Ralph 这类外部 loop 方案，它的核心思路就是反复运行 AI coding agent，让每轮保持干净上下文，同时通过 git 和状态文件保持长期记忆。

## 四、8 小时任务要搭配哪些 harness？

如果拆开看，一套能支撑长程任务的 harness，通常至少包括五层：循环层、状态层、工具层、验证层和安全层。

![Harness 五层架构图](whiteboard-1.png)

| 层级 | 作用 | 典型实现 |
|------|------|----------|
| **Loop 层（循环层）** | 让模型持续执行，不是一轮结束就停 | Claude Code 的 `/loop`、Ralph 等外部 loop 方案 |
| **State 层（状态层）** | 把目标、待办、当前状态、失败记录写到文件里 | CLAUDE.md、外部进度文件（goal.md / todo.md / progress.md） |
| **Tool 层（工具层）** | 让模型能读写文件、跑命令、调用外部服务 | Claude Code 内置工具 + MCP 扩展 |
| **Eval 层（验证层）** | 告诉系统"这轮是不是有效推进了" | lint、test、benchmark、脚本检查 |
| **Safety 层（安全层）** | 权限、沙箱、回滚、关键节点人工介入 | permissions、hooks、断路器 |

这个拆法和 Claude Code 官方提供的扩展能力也非常接近：CLAUDE.md 负责持久上下文，hooks 在生命周期关键节点自动执行，MCP 用来连接外部工具和数据源，subagents 用来拆分任务和隔离上下文，Agent SDK 则把整套 agent loop 和上下文管理能力程序化暴露出来。

## 五、用一个案例理解：一项 8h 长程任务是怎样跑出来的？

如果要跑一个"从零搭建 Linux 桌面系统"的长程任务，起手通常不是一句"做一个 Linux 桌面"，而是先把任务收敛成一个用户态的 Linux 桌面原型：例如基于 Python + PyQt6 或 GTK，先实现桌面主界面、窗口管理体验、状态栏、启动器、基础应用、设置面板和字体支持，再逐步补 VPN 管理和游戏入口。

### 预先准备

可以在项目根目录先放这几个文件：

| 文件名 | 目的 | 要点举例 |
|--------|------|----------|
| CLAUDE.md | 写长期规则，告诉它这次任务怎么推进。 | 先规划，再实现，不要一上来铺开所有模块；每完成一个阶段都更新 todo.md 和 progress.md；每轮只推进一个核心子目标；每做完一段实现就运行对应检查；非核心功能不要提前展开 |
| goal.md | 写清目标和边界。 | 目标：做一个可本地运行的 Linux 桌面原型；必须包含：桌面主界面、顶部状态栏、应用启动器、窗口管理体验、设置页、中文字体支持、基础应用；可选项：VPN 管理、游戏入口；非目标：内核、驱动、真正的底层 compositor 重写 |
| acceptance.md | 写清验收标准。 | 程序能在本地启动；能打开和关闭至少 3 个应用窗口；状态栏能显示时间和基本状态；启动器能打开应用；中文文本显示正常；有 README 和启动方式 |
| todo.md / progress.md | 一个写阶段任务，一个写每轮进展和遗留问题。 | |

### 第一轮提示词应该怎么写

第一轮要让它先完成方案收敛 + 骨架搭建 + 最小版本跑通。

示例如下：

> 请把当前仓库作为一个从零开始的 Linux 桌面原型项目来处理。
>
> 先阅读并遵守以下规则：
> 1. 这是一个长程工程任务，不要试图一次完成全部功能。
> 2. 先做规划和骨架，再做最小可运行版本，再逐步补模块。
> 3. 每完成一个阶段，都更新 todo.md 和 progress.md。
> 4. 每一轮只推进一个核心子目标，并在完成后运行必要的验证命令。
> 5. 不要提前展开非核心功能。
>
> 任务目标：
> 构建一个可本地运行的 Linux 桌面原型，优先实现：
> - 桌面主界面
> - 顶部状态栏
> - 应用启动器
> - 基础窗口管理体验
> - 设置页
> - 中文字体支持
>
> 可选功能：
> - VPN 管理入口
> - 游戏入口
>
> 请按以下顺序执行：
> A. 先给出技术选型建议，并在 Python + PyQt6 / GTK 中选一个更适合快速落地的方案
> B. 生成项目目录结构和阶段计划
> C. 创建 CLAUDE.md、goal.md、acceptance.md、todo.md、progress.md
> D. 只实现 Phase 1：最小可运行版本
> E. 运行项目并汇报当前结果、已知问题、下一阶段建议
>
> 注意：
> - 当前回合不要尝试完成全部模块
> - 当前回合的目标是得到一个能启动的最小桌面原型
> - 输出时请明确说明：已完成什么、未完成什么、下一步做什么

### 第一轮跑完后，第二轮怎么接

等它把最小版本跑起来后，第二轮再收窄目标，不要继续说"往下做"。你要指定一个模块，让它围绕单一子目标推进。比如：

> 继续当前项目，只处理 Phase 2，不要扩展到其他阶段。
>
> 本轮目标：
> 1. 完善顶部状态栏，显示时间、网络状态占位和系统菜单入口
> 2. 实现一个简单的应用启动器，可启动至少 3 个内置应用窗口
> 3. 保持现有最小桌面原型可运行
>
> 执行要求：
> - 先阅读 todo.md 和 progress.md，确认当前状态
> - 修改后运行项目验证
> - 更新 todo.md 和 progress.md
> - 输出本轮改动摘要、运行结果、遗留问题

这样写的好处是，它不会在第二轮又重新铺开整个系统，而是继续沿着当前工程状态前进。

后续几轮的推进方式也基本类似：每一轮都围绕一个明确的子目标展开，在当前工程状态上继续实现、验证、修复和记录，而不是反复把任务重新铺开。

当桌面主界面、状态栏、启动器、设置页等核心模块逐步补齐后，任务才会进入下一阶段。

### 到中后段，提示词要从"做功能"切到"做集成"

很多长程任务跑散，不是因为前面做不出来，而是中后段还在不停加功能，没有及时切到集成和修复。

所以当桌面、状态栏、启动器、设置页这些模块差不多都有了以后，提示词也要跟着切换。比如：

> 从现在开始，任务重点切换为集成与收敛，不再新增大功能。
>
> 本轮目标：
> 1. 检查桌面主界面、状态栏、启动器、设置页之间的交互是否正常
> 2. 修复模块之间的依赖和启动问题
> 3. 检查中文字体显示是否正常
> 4. 补 README 中的启动说明和项目结构说明
>
> 执行要求：
> - 优先修复可运行性和集成问题
> - 不新增非必要模块
> - 完成后运行验证，并更新 progress.md

这一步很重要。因为前半段是"把东西做出来"，后半段是"把东西接起来"。两种提示词不能混着写。

到这里，这项 8h 长程任务的推进路径就比较清楚了：前期先搭骨架，中期逐步补模块，后期重点做集成、修复和收敛。

## 六、开发者真要上手，建议从什么配置开始？

如果你是第一次尝试长程任务，不需要一上来就搭很重的系统。比较推荐的做法，是先从一个最小可用配置开始：一个能稳定执行长程任务的模型、一套能循环调度的执行器、一组外部状态文件、最基本的命令行工具和测试脚本，再加上几个关键节点的人工 review。

Claude Code 官方已经把很多基础设施准备好了，包括 agentic coding 的执行方式、/loop、CLAUDE.md、hooks、MCP 和 Agent SDK；如果你更希望把任务做成"每轮独立、外部状态持久化"的方式，Ralph 这类 loop 方案会更贴近那种连续自动迭代的工作流。

一个最小可用栈，大概可以长这样：

| 模型 | GLM-5.1 | 作为核心执行者，负责理解任务、拆解步骤、调用工具并持续推进。 |
|------|---------|----------------------------------------------------------|
| 执行器 | 内置 loop 或外部 loop | 负责让任务不是一轮结束就停止，而是能够按设定节奏持续运行，把多个执行回合串起来。 |
| 状态文件 | goal.md、todo.md、progress.md | 用来把任务目标、待办事项和每轮进展写到外部，而不是全部依赖上下文记忆。例如：goal.md 写清最终目标和约束，todo.md 维护当前任务拆解，progress.md 记录每轮完成了什么、还剩什么问题。 |
| 工具 | shell、git、测试脚本 | 用来读写文件、执行命令、查看结果和保存阶段性修改，保证模型不是只"输出建议"，而是真正能动手推进任务。 |
| 验证方式 | 单测、lint、run script、benchmark | 用来判断当前修改是否有效，避免任务长时间运行后偏离目标。实践里不一定每一步都跑完整验证，但至少每完成一段核心改动，就要有对应检查。 |
| 人工介入 | 阶段性 review | 不需要全程盯着模型执行，但建议在关键节点介入，例如：最小版本跑通后看一次，核心功能补齐后看一次，最终交付前再统一检查一次。 |

---

回到最开始那个问题：GLM-5.1 的 8 小时长程任务到底意味着什么？对开发者来说，它真正带来的变化，是我们终于可以把一类原本需要人不断盯着推进的复杂工程任务，交给一个更稳定的执行闭环去持续完成。 这类能力当然建立在模型本身之上，但真正决定任务能不能跑成的，往往不是模型单点性能，而是 loop、状态管理、工具接入、验证机制和安全边界有没有搭好。

所以，理解长程任务，最重要的一步不是问"它能不能自己干 8 小时"，而是问：我有没有给它搭一套，值得它连续工作 8 小时的系统。
